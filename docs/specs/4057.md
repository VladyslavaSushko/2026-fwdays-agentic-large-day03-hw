# Issue #4057 — Can't Rotate to 90° in Grid Mode

**Source:** https://github.com/excalidraw/excalidraw/issues/4057
**Status:** Open | **Labels:** bug, good first issue

## Goal
When grid mode is enabled, snap rotation to 90° increments automatically —
without requiring the user to hold Shift.

## Context
- Issue: #4057
- Current behavior: rotating an element in grid mode does not snap to 90°;
  pointer coordinates are grid-snapped, which prevents hitting exact angle multiples
- Expected: rotation snaps to 0°, 90°, 180°, 270° when grid mode is on

## Component / Module
- `packages/element/src/resizeElements.ts` — `rotateSingleElement`, `rotateMultipleElements`, `transformElements`
- `packages/excalidraw/components/App.tsx` — `transformElements` call site (~line 12424)

## Non-goals
- Do NOT change Shift-key angle locking behavior (15° increments)
- Do NOT affect resize logic — only rotation
- Do NOT modify grid rendering or grid size settings
- Do NOT change rotation behavior when grid mode is off

## Acceptance Criteria
1. Rotating an element in grid mode snaps to 0°, 90°, 180°, 270°
2. Holding Shift still snaps to 15° increments regardless of grid mode
3. Rotating without grid mode and without Shift remains free-form
4. Existing tests pass (`yarn test`)
5. Build passes (`yarn build`)

## Edge Cases
- **Shift + grid mode simultaneously** — Shift takes priority (`else if` ordering); snaps to 15° not 90°
- **Frame elements** — always locked to 0°, grid snap is never reached (`isFrameLikeElement` check before angle calc)
- **Elbow arrows** — excluded from rotation entirely (`isElbowArrow` guard in `transformElements`)
- **Multiple elements** — `rotateMultipleElements` applies the same `GRID_ROTATION_SNAP_ANGLE` to the group center angle, consistent with single element behavior

---

### Root Cause

The rotation pointer coordinates are implicitly snapped to grid points. Because the grid snaps X/Y to integer multiples of `gridSize`, the computed angle:

```
angle = (5π/2) + atan2(pointerY - cy, pointerX - cx)
```

never hits an exact multiple of `SHIFT_LOCKING_ANGLE` (15°), so discrete-angle snapping (Shift key) cannot snap to 90°, 180°, etc.

The discrete angle logic lives in `rotateSingleElement` and `rotateMultipleElements`:

```ts
// packages/element/src/resizeElements.ts:222-224
if (shouldRotateWithDiscreteAngle) {
  angle = (angle + SHIFT_LOCKING_ANGLE / 2) as Radians;
  angle = (angle - (angle % SHIFT_LOCKING_ANGLE)) as Radians;
}
```

`shouldRotateWithDiscreteAngle` is **only** `event.shiftKey` (`packages/common/src/keys.ts:149`). Grid mode has no effect on rotation snapping at all — that is the bug.

## Blast Radius

**Small.** The change is confined to the rotation code path only:

- `transformElements` is called from two places in `App.tsx` (lines ~12424 and nowhere else for rotation).
- `rotateSingleElement` and `rotateMultipleElements` are private functions inside `resizeElements.ts`.
- No element data model changes — only the computed `angle` value is affected.
- No rendering, history, binding, or snapping infrastructure changes needed.
- Existing Shift-key behaviour is **preserved** — the new grid snap is additive, not replacing.

The only behavioural risk: users who rely on free-form rotation while grid mode is on will now be snapped to 90° increments. This matches user expectation (grid = precision) and is consistent with how grid affects position dragging.

## Implementation Plan

### Step 1 — Extend `transformElements` signature

In `packages/element/src/resizeElements.ts`, add a `gridModeEnabled: boolean` parameter:

```ts
export const transformElements = (
  originalElements: PointerDownState["originalElements"],
  transformHandleType: MaybeTransformHandleType,
  selectedElements: readonly NonDeletedExcalidrawElement[],
  scene: Scene,
  shouldRotateWithDiscreteAngle: boolean,
  shouldResizeFromCenter: boolean,
  shouldMaintainAspectRatio: boolean,
  pointerX: number,
  pointerY: number,
  centerX: number,
  centerY: number,
  gridModeEnabled = false,   // ← new
): boolean
```

Pass it down to `rotateSingleElement` and `rotateMultipleElements`.

### Step 2 — Apply 90° snap in `rotateSingleElement`

`GRID_ROTATION_SNAP_ANGLE` is defined in `packages/common/src/constants.ts` alongside `SHIFT_LOCKING_ANGLE`.

```ts
// packages/element/src/resizeElements.ts ~line 226
if (shouldRotateWithDiscreteAngle) {
  angle = (angle + SHIFT_LOCKING_ANGLE / 2) as Radians;
  angle = (angle - (angle % SHIFT_LOCKING_ANGLE)) as Radians;
} else if (gridModeEnabled) {
  angle = (angle + GRID_ROTATION_SNAP_ANGLE / 2) as Radians;
  angle = (angle - (angle % GRID_ROTATION_SNAP_ANGLE)) as Radians;
}
```

Apply the same logic in `rotateMultipleElements` (~line 426).

### Step 3 — Update the caller in `App.tsx`

At line ~12424, pass the grid flag:

```ts
transformElements(
  pointerDownState.originalElements,
  transformHandleType,
  selectedElements,
  this.scene,
  shouldRotateWithDiscreteAngle(event),
  shouldResizeFromCenter(event),
  ...,
  isGridModeEnabled(this),   // ← new last argument
)
```

`isGridModeEnabled` is already imported from `packages/excalidraw/snapping.ts`.

### Step 4 — Tests

Add test cases in `packages/excalidraw/tests/` (e.g. `resizeElements.test.ts`):

- Rotating an element in grid mode snaps to 0°, 90°, 180°, 270°.
- Rotating with Shift still snaps to 15° increments (existing behaviour unchanged).
- Rotating without Shift and without grid is free-form (existing behaviour unchanged).
